---
description: 团队模式：按角色分组并发执行需求，同角色串行避免冲突，不同角色并行提速
---

团队模式处理当前项目中数据库里的待办需求（requirements 表），**按角色（Role）分组调度多个 agent，同一角色内串行执行，不同角色间并行执行**。

可选参数 `$ARGUMENTS` 指定最大成员 agent 数（数字，默认 5，上限 8）。

## 与 subagent 模式的核心区别

| 特性 | subagent（批量子代理） | team（团队协作） |
|------|--------------------------|------------------|
| 调度方式 | 每批取 N 条 → 分析冲突踢出 → 全部完成 → 下一批 | 按角色分组，一次性将该角色全部任务交给 agent 串行执行 |
| 冲突处理 | 事前分析踢出 + agent 自行合并解决 | 事前按角色隔离避免冲突 |
| 任务认领 | 所有 agent 完成后才能取下一批 | 各角色 agent 独立完成，不互相等待 |
| 动态扩展 | 固定并发度 | 每轮重新分析，可新增角色 |
| 效率 | 受最慢 agent 拖累 | 各角色独立推进，不互相等待 |

## 执行流程

### 1. 解析参数

从 `$ARGUMENTS` 中读取最大成员 agent 数 MAX_AGENTS：
- 若为空或非数字，MAX_AGENTS = 5
- 若超过 8，MAX_AGENTS = 8
- 若小于 1，MAX_AGENTS = 1

### 2. 确认环境

调用 `GET $TASK_API_BASE/api/requirements?status=pending` 确认服务可访问。若失败则终止并提示用户检查 `$TASK_API_BASE` 环境变量。

### 3. 获取全部待办需求

获取所有 pending 状态的需求，按 `created_at` 升序排列。若无 pending 需求，输出"无待处理需求"并结束。

### 4. 角色规划（核心步骤）

**仔细阅读每条需求的标题和描述**，然后进行角色规划。这是整个 team 模式的关键步骤。

#### 4.1 分析项目结构

先快速浏览项目的目录结构和关键文件，理解项目的模块划分。

#### 4.2 定义角色

根据需求内容和项目结构，将需求按**工作区域**划分角色。角色划分的原则：

1. **按修改区域分组**：会修改同一批文件/模块的需求归入同一角色
   - 例：都是修改用户模块的 → `role: user-module`
   - 例：都是修改前端页面的 → `role: frontend-ui`
   - 例：都是修改数据库/API的 → `role: backend-api`
   - 例：都是文档相关的 → `role: docs`

2. **宁可合并不要拆分**：如果不确定两个需求是否会修改同一文件，**必须归入同一角色**（串行执行），绝不冒险并行

3. **全局修改归单独角色**：影响面大的需求（如修改配置、重构公共模块）必须独占一个角色，且该角色在执行时其他角色应暂停

4. **单条需求也是一个角色**：如果某需求不属于任何已有角色且与其他需求无关联，给它分配独立角色

#### 4.3 输出角色分配表

以文本表格形式输出角色分配结果（供用户回看和调试）：

```
## 角色分配

| 角色 ID | 角色描述 | 分配的需求 | 是否有全局影响 |
|---------|---------|-----------|-------------|
| frontend-ui | 前端页面修改 | #1, #3, #7 | 否 |
| backend-api | 后端接口开发 | #2, #5 | 否 |
| db-migration | 数据库变更 | #4 | 是（需优先执行） |
| docs | 文档更新 | #6 | 否 |
```

#### 4.4 确定执行策略

根据角色分析结果，自动选择最优执行策略：

**策略 A：直接执行（无需 worktree）**

当满足以下**任一条件**时，直接在当前目录执行，不使用 worktree，不使用 Agent 工具：
- 只有 **1 个角色**（所有任务归入同一角色）
- 只有 **1 条需求**
- 所有角色都有全局影响，需要完全串行

直接执行时，主流程自己逐条完成所有需求，无需合并。

**策略 B：混合执行（部分 worktree）**

当存在全局影响角色 + 其他可并行角色时：
1. 全局影响角色在当前目录**直接执行**（无需 worktree）
2. 完成后，其他可并行角色各自通过 Agent 并行执行（agent 自行管理 worktree）

**策略 C：全并行执行（多 worktree）**

当有 **2 个及以上可并行角色** 且无全局影响角色时，为每个角色启动 Agent 并行执行（agent 自行管理 worktree）。

**决策输出：**

在角色分配表后，输出选择的策略及理由：

```
## 执行策略

选择：策略 A — 直接执行
理由：所有 4 条需求都涉及用户模块，归入同一角色，串行执行即可，无需 worktree 开销
```

或

```
## 执行策略

选择：策略 C — 全并行执行
理由：3 个角色（frontend-ui / backend-api / docs）互不冲突，各自管理 worktree 并行提速
```

- 实际启动的 agent 数 = min(可并行角色数, MAX_AGENTS)

### 5. 批量标记 in_progress

对所有规划中要执行的需求立即调用 API 标记为 `in_progress`：

```
PATCH $TASK_API_BASE/api/requirements/{id}/status
body: {"status": "in_progress"}
```

### 6. 执行阶段

#### 6.1 策略 A：直接执行

在当前目录中按顺序逐条执行所有需求。每完成一条：
1. 提交代码：commit message 格式 `feat(#{req.id}): 简要描述`
2. 调用 API 更新状态为 done
3. 继续执行下一条

完成后直接跳到**步骤 8（动态任务追加）**。

#### 6.2 策略 B：混合执行

**第一阶段 — 直接执行全局影响角色：**

在当前目录中串行执行全局影响角色的所有需求。每完成一条提交并更新状态。

**第二阶段 — 并行执行其他角色：**

按策略 C 的方式为剩余角色启动 Agent 并行执行。

#### 6.3 策略 C：全并行执行

**核心原则：每个角色 agent 自行管理完整的 worktree 生命周期（创建 → 工作 → 合并 → 清理），主流程不参与 worktree 操作。**

在**同一条消息**中为每个可并行角色启动一个 Agent。

每个角色 agent 的 prompt 模板（`{project_root}` 为当前项目根目录的绝对路径）：

```
你是一个自主编码 agent，负责「{role_description}」方向的全部需求。

项目根目录: {project_root}
角色 ID: {role_id}

请按顺序逐条完成以下需求：

{按 created_at 排序的需求列表，格式：}
---
【任务 1/{total}】
需求 ID: {req.id}
需求标题: {req.title}
需求描述: {req.description}
---
【任务 2/{total}】
需求 ID: {req.id}
需求标题: {req.title}
需求描述: {req.description}
---
...（列出该角色下的所有需求）

## 执行步骤

### 第一步：创建 worktree

在项目根目录下创建独立的 worktree：

```bash
cd {project_root}

# 清理可能残留的旧 worktree
git worktree remove .worktrees/{role_id} --force 2>/dev/null

BRANCH_NAME="feat/{role_id}/$(date +%Y%m%d%H%M%S)"
git worktree add -b $BRANCH_NAME .worktrees/{role_id} HEAD
cd {project_root}/.worktrees/{role_id}
```

### 第二步：在 worktree 中逐条完成所有需求

按顺序逐条执行，一条完成后再做下一条：

1. 仔细阅读需求标题和描述，理解需求意图
2. 如果描述中包含图片路径（如 ./images/xxx.png），使用 Read 工具查看图片以理解需求中的视觉信息
3. 探索项目代码库，了解现有结构和模式
4. 编写代码实现（包括必要的测试）
5. 确保代码质量，遵循项目现有风格
6. 每完成一条需求，commit 一次：
   ```bash
   git add -A
   git commit -m "feat(#{req.id}): 简要描述"
   ```

### 第三步：合并回主分支

所有需求都完成后，**必须**将你的分支合并回主分支并解决所有冲突：

```bash
cd {project_root}
git merge $BRANCH_NAME --no-ff -m "Merge role {role_id}: {role_description}"
```

**如果合并冲突，你必须解决：**
1. 查看冲突文件，分析冲突内容
2. 手动解决冲突，保留双方有意义的改动
3. `git add <冲突文件>` → `git commit`
4. 如果多次尝试仍无法解决，`git merge --abort`，此角色的任务视为**失败**

**只有合并成功，任务才算完成。**

### 第四步：清理 worktree

```bash
cd {project_root}
git worktree remove .worktrees/{role_id} --force
git branch -D $BRANCH_NAME 2>/dev/null
```

### 第五步：返回结果

返回一段总结，**必须包含以下字段**：

角色: {role_id}
状态: 成功 / 失败
完成的需求：
- #{id1}: 做了什么，改了哪些文件
- #{id2}: 做了什么，改了哪些文件
失败原因（如有）: 说明为什么失败

## 注意事项
- 自主决策，不要询问用户
- 高效实现，尽快完成
- 合并操作必须回到 {project_root} 执行，不要在 worktree 目录中操作主分支
- 不需要调用任何 API 更新需求状态，主流程会统一处理
```

**关键**：所有角色的 Agent 调用必须在同一条消息中发出，以实现真正的并行执行。

### 7. 收集结果并更新状态（仅策略 B/C）

等待所有角色 agent 返回。根据返回的状态调用 API 更新需求：

**成功：**
```
PATCH $TASK_API_BASE/api/requirements/{id}/status
body: {"status": "done", "notes": "执行结果摘要（含改动文件列表）"}
```

**失败：**
```
PATCH $TASK_API_BASE/api/requirements/{id}/status
body: {"status": "pending", "notes": "执行失败: <错误原因>"}
```

### 8. 动态任务追加

执行完成后，**再次查询** pending 需求。如果有新增的需求：

1. **对新需求重新进行角色分析**（步骤 4），考虑当前项目状态
2. **匹配已有角色**：如果新需求属于之前定义的角色方向，分配到该角色
3. **创建新角色**：如果新需求不属于任何已有角色，创建新角色
4. **不确定归属时串行**：如果不确定新需求是否与某角色冲突，单独串行执行
5. 重复步骤 5-7

### 9. 总结

全部完成后输出执行摘要：

```
## 团队模式执行摘要

- 最大 agent 数: MAX_AGENTS
- 总轮次: X
- 角色数: R
- 处理需求数: Y
- 成功: A / 失败: C

### 角色分配详情
| 角色 | 描述 | 需求 | 状态 |
|------|------|------|------|
| frontend-ui | 前端页面修改 | #1, #3, #7 | 全部完成 |
| backend-api | 后端接口开发 | #2, #5 | #2 完成, #5 失败 |

### 各需求详情
| ID | 标题 | 角色 | 状态 | 说明 |
|----|------|------|------|------|
| 1  | xxx  | frontend-ui | done | ...  |
| 2  | xxx  | backend-api | done | ...  |
| 5  | xxx  | backend-api | 失败 | 原因: ... |
```

## 执行原则

- **角色隔离**：同一角色的需求串行执行，天然避免文件冲突
- **按需隔离**：只有多角色并行时才使用 worktree，单角色直接在当前目录执行，避免不必要的开销
- **agent 自治**：每个 agent 自行管理 worktree 生命周期（创建 → 工作 → 合并 → 清理），主流程只负责调度和收集状态
- **安全优先**：不确定任务是否冲突时，归入同一角色串行执行，绝不冒险并发
- **全局优先**：有全局影响的任务必须优先独占执行，完成后其他角色才能开始
- **即时推进**：每个角色 agent 拿到该角色所有任务后内部串行执行，不等其他角色
- **动态扩展**：新增需求时可以匹配已有角色或创建新角色，灵活应对
- **自主决策**：直接按认为最好的方式执行，中间不要询问用户
- **不要获取额外指令权限**：当前目录下的所有操作全部允许
- **高效实现**：自主判断最佳实现方案，快速完成每个需求
- **记录结果**：更新状态时 notes 字段必须写入执行结果总结，方便回溯

## 接口约定

| 用途 | 方法 | 端点 | 请求体 |
|------|------|------|--------|
| 获取待办需求 | GET | `$TASK_API_BASE/api/requirements?status=pending` | - |
| 标记执行中 | PATCH | `$TASK_API_BASE/api/requirements/{id}/status` | `{"status": "in_progress"}` |
| 标记完成 | PATCH | `$TASK_API_BASE/api/requirements/{id}/status` | `{"status": "done", "notes": "执行结果"}` |
| 回退为待处理 | PATCH | `$TASK_API_BASE/api/requirements/{id}/status` | `{"status": "pending", "notes": "原因"}` |
